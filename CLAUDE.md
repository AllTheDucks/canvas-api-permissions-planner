# Canvas API Permissions Planner — Project Context

## Purpose

A fully client-side React/TypeScript tool for Canvas LMS extension developers. Users enter the Canvas REST API endpoints their extension calls, and the tool tells them which role permissions the API user must have in Canvas.

## Tech Stack

| Tool | Version | Notes |
|------|---------|-------|
| Vite | 6 | Build tool and dev server |
| React | 18 | UI framework |
| TypeScript | 5 | Strict mode enabled |
| Mantine | 7 | Component library — uses CSS Modules + PostCSS (not Emotion) |
| Zod | 3 | Schema validation for ALL externally loaded data |
| js-yaml | 4 | Parses Canvas YAML locale files fetched from GitHub |
| pnpm | latest | **Always use pnpm. Never npm or yarn.** |

## Key Architectural Decisions

- **No server.** Everything runs in the browser. No SSR, no API proxy, no build-time data fetching. No backend database.
- **No application-state persistence.** Selected endpoints, SIS options, and all working state reset on page reload. `localStorage` is used only for UI preferences (color scheme, locale override) — not for application data.
- **Validate external data with Zod.** Any data loaded from outside the bundle (`endpoints.json` fetched at runtime, Canvas locale YAML from GitHub) must be validated through a Zod schema before use.
- **Static hosting target.** The app is built with `pnpm build` and served as static files. GitHub Pages is the intended host, with a custom domain (`canvas-permissions.alltheducks.com`). Vite `base` stays at the default `'/'`.

## Data Source

`public/data/endpoints.json` is the canonical permission mapping data, served as a static asset and fetched at runtime (not bundled). It is generated by Claude reading the Canvas LMS source code on GitHub — see `docs/regenerate-data-prompt.md` for the full prompt. It contains:
- A `permissions` map: symbol → `{ label, scope }` (English labels)
- An `endpoints` array: all ~300 API endpoints with their required permission symbols

See `plans/design.md` for the full JSON shape and Zod schemas.

## Localisation

Canvas permission labels use `I18n.t("Grades - edit")` inline in Ruby (via i18nliner), not symbolic YAML keys. This means:
- English labels are hard-coded in `endpoints.json` and always available offline
- For other locales, `useLocale.ts` fetches the target locale YAML from `raw.githubusercontent.com/instructure/canvas-lms/master/config/locales/{locale}.yml`
- The YAML key is derived at runtime from the English label using the i18nliner `keyify_underscored_crc32` algorithm: `{slug}_{CRC32("{len}:{label}").toString(16)}` — e.g. `"Grades - edit"` → `grades_edit_773dfc24`
- Falls back to English for any permission label not found in the target locale
- Fetched locales are cached in memory for the session

See `plans/design.md` — "Localisation Architecture" section for full details.

## Project Structure Overview

```

├── data/endpoints.json          # Permission mapping data (fetched at runtime)
src/
├── schemas/                     # Zod schemas (endpoints.ts, canvasLocale.ts)

├── hooks/                       # useEndpoints, useLocale

```

## Coding Conventions

- Components live in their own subdirectory with an `index.tsx` file
- No default exports from utility files — named exports only
- Prefer functional components with hooks
- Mantine v7 component imports come from `@mantine/core`
- Do not add comments or docstrings to code unless the logic is genuinely non-obvious

## Mantine v7 Notes

- Requires PostCSS: `postcss.config.cjs` must include `postcss-preset-mantine` and `postcss-simple-vars`
- CSS import required in `main.tsx`: `import '@mantine/core/styles.css'`
- No Emotion dependency (v7 dropped it)
- Wrap the app root in `<MantineProvider>` in `main.tsx`

## Do Not

- Add server-side code or Node.js runtime dependencies
- Add databases or authentication
- Use npm or yarn (always pnpm)
- Fetch data at build time
- Add features not listed in `todo.md` or requested by the user
- Add error handling for scenarios that cannot happen
